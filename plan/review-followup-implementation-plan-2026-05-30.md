# investor レビュー反映 実装計画

作成日: 2026-05-30  
対象: `1_now/investor/`  
ベース文書: `docs/review-summary-2026-05-30.md`

---

## この計画の目的

今回のレビューで見えた論点は多いが、全部を同時に直すと軸がぼける。  
この計画では、次の順で進める。

1. まず壊れ方が危ないところを止血する
2. 次にデータ契約をそろえる
3. その後で校正ループを主役に押し上げる
4. 最後に B枠と仮説検証を回し始める

つまり、**運用の信頼性 → 計測の信頼性 → 学習ループ** の順。

---

## 実装方針

### 主目的

`investor/` を「毎日提案を出すツール」から、  
**判断を記録し、検証し、次回に反映する投資判断OS** に寄せる。

### 今回の非目標

今回は次をやらない。

- 新しい UI の実装
- debate ペルソナの追加
- リアルタイム監視の高度化
- 新しい外部APIの導入
- 戦略そのものの全面変更

---

## 優先順位

| 優先度 | テーマ | 目的 |
|---|---|---|
| P0 | 信頼境界の修正 | 壊れる経路を止める |

## Phase P0: 信頼境界の修正

### P0-1. Slack 依存を送信時だけに限定する

**対象**

- `investor/config.py`
- `investor/notifications/slack.py`
- `investor/agents/decision_agent.py`
- `skills/decision.py`

**やること**

- `Settings()` の import 時点で Slack 必須にならないようにする
- 読み取り専用経路と `--paper` 経路では Slack 設定がなくても動くようにする
- Slack が必要なのは実際に送信するタイミングだけに絞る

**完了条件**

- `.env` がなくても `/decision` 読み取り経路が動く
- `.env` がなくても `--paper` が動く
- Slack 未設定で送信経路だけが明示的に失敗する

### P0-2. 同一 ticker の追加買い制約を入れる

**対象**

- `investor/agents/decision_agent.py`
- `tests/test_skills.py`

**やること**

- `validate_proposals()` で既存 open position を含めて判定する
- 同一 ticker をすでに持っている場合、追加 BUY をどう扱うかルール化する

候補ルール:

1. 追加 BUY を全面禁止
2. 追加 BUY は許可するが、合計エクスポージャが 25% を超えたら拒否

この計画では 2 を採用する。  
理由は、今後段階的な追加エントリー余地を残しつつ、README の制約とも整合するため。

**完了条件**

- 同一 ticker の既存保有ありで 25% 超過ならブロックされる
- 既存保有なしなら従来通り判定される
- テストで再現できる

### P0-3. 表示文言を実態に合わせる

**対象**

- `scripts/weekly_review.py`
- 関連 README / docs

**やること**

- `BUY実行` を `BUY提案` へ変更するか、実約定ログを別集計に切る
- 現段階ではまず誤読を防ぐため、表記を `BUY提案` に寄せる

**完了条件**

- 週次レビューを読んだ人が「人間が約定した数」と誤解しない

**完了条件**

- 集計処理が silent skip ではなく、必要なら warning を出す

### P1-3. `entry_price_range` パースの統一

**対象**

- `investor/agents/decision_agent.py`
- `skills/decision.py`

**やること**

**完了条件**

- `"150-160"` と `"150–160"` で同じ結果になる
- `shares_suggested` が静かに `None` にならない

## Phase P2: 提案ログと実約定ログの分離

### P2-1. ログの役割を分ける

**現状**

- `decision_history.json` は提案履歴
- `portfolio.csv` は実約定履歴
- しかしレビュー側では両者が混ざっている

**方針**

- `decision_history.json` は「AIが何を提案したか」
- `portfolio.csv` は「人間が何を執行したか」

として明確に分ける

### P2-2. 週次レビューの集計を二層化する

**対象**

- `scripts/weekly_review.py`

**やること**

- 提案サマリー
- 実約定サマリー

を分けて表示する

最低限ほしい項目:

- 提案 BUY 数
- 提案 PASS 数
- 実約定数
- 実クローズ数
- 実現損益
- SPY alpha

**完了条件**

- 「提案の質」と「執行後の結果」が分離される
- 週次レビューの数字の意味が明確になる

## Phase P3: 校正ループの昇格

### P3-1. `show_calibration_stats.py` を判断前の標準入力にする

**対象**

- `scripts/show_calibration_stats.py`
- `investor/agents/decision_agent.py`
- `skills/research.py`

**やること**

- いまは参考表示に近いので、判断前コンテキストとして位置づける
- `HIGH / MEDIUM / LOW` の勝率だけでなく、alpha を主役にする

**見るべき指標**

- conviction ごとの平均 alpha
- signal_type ごとの平均 alpha
- 保有日数ごとの傾向
- recent EV の劣化有無

### P3-2. PASS の質も追う

**やること**

- BUY の勝敗だけでなく、PASS した銘柄がその後どうなったかを見る余地を残す
- すぐ全部はやらなくてよいが、少なくとも設計上は PASS を死データにしない

### P3-3. `no trade` を正当な出力にする

**背景**

- 毎回候補を出し、毎回 debate する設計だと、エントリー圧力が高くなる

**やること**

- `HOLD_CASH`
- `NO_TRADE`

を正規の判断として扱いやすくする

**完了条件**

- 「何もしない」が失敗ではなく、明示的な判断として残る

## Phase P4: B枠運用の立ち上げ

### P4-1. paper portfolio を実験枠として明確化する

**対象**

- `skills/paper_portfolio.py`
- `skills/decision.py`
- docs

**やること**

- B枠は「Slack 送信しない代替経路」ではなく、仮説検証枠だと明文化する
- real と paper の違いを週次で比較できるようにする

### P4-2. 比較指標を決める

最初はこれで十分:

- 平均リターン
- 勝率
- 平均 alpha
- 保有日数
- 提案から執行までの差

### P4-3. 最初の仮説を 1 つだけ回す

候補:

1. watchlist 経由のほうが成績が良いか
2. すぐ入るより押し目待ちのほうが良いか
3. HIGH conviction の sizing は大きすぎるか

今回は 1 か 2 のどちらかに絞る。  
複数同時に回すと何が効いたかわからなくなる。

## 実装時の注意

### 1. 既存データを壊さない

- `portfolio.csv`
- `research_history.json`
- `decision_history.json`

はすでに履歴として価値がある。  
列追加や名称変更をする場合は、読み取り互換を先に入れてから移行する。

### 2. ドキュメントとコードを一緒に直す

今回の違和感の一部は、コードより README / docs の説明ズレだった。  
特に次はコード変更と同時に更新する。

- `README.md`
- `docs/workflow-overview.md`
- 必要なら `docs/review-summary-2026-05-30.md`

### 3. テストを後回しにしない

少なくとも次のケースはテスト化する。

- Slack 未設定で read-only 経路が動く
- 同一 ticker の追加 BUY 制約
- en dash / hyphen の価格帯パース
- closed row 欠損時の扱い
- weekly review の提案数 / 実約定数の分離

---

## 完了の定義

この計画が一段落したと言える条件は次の通り。

1. Slack 未設定でも読み取り系が壊れない
2. 1銘柄 25% 制約がコード上でも効いている
3. review の数字が「提案」と「実約定」で混ざらない
4. CSV / JSON 契約の主要なズレが解消されている
5. 校正レポートを次回判断に返す流れが明確になっている
6. B枠を少なくとも 1 つの仮説検証枠として動かし始めている

---

## 最後に

今回のレビューでわかったことは、`investor/` に足りないのは派手な新機能ではない、ということだった。

必要なのは次の3つ。

1. 壊れ方を減らす
2. 数字の意味を揃える
3. 判断の学習ループを中心に置く

この順で進めれば、`investor/` は「その場で銘柄を勧めるAI」から、  
**自分の投資判断を継続的に鍛えるシステム** に近づく。
