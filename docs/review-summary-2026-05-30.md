# investor レビューまとめ

作成日: 2026-05-30  
対象: `1_now/investor/`  
範囲: コードレビュー + 戦略レビュー + 既存レビュー文書の突合

---

## 先に結論

`investor/` はすでに「動く投資補助ツール」にはなっている。  
ただし、現状の強みは売買提案そのものより、**投資判断を記録し、あとで検証できる土台を持っていること**にある。

一方で、今は次の2つが混ざっている。

1. 毎日の売買判断を出す運用ツール
2. 判断の質を学習して改善する校正システム

この2つがまだ整理されていないため、Slack 通知、paper portfolio、weekly review、watchlist、decision debate がそれぞれ部分最適になっている。

いま優先すべきは、機能追加より **信頼境界の修正** と **評価ループの明確化**。

---

## 今回のレビューで見えた全体像

### 良い点

- `Claude Code IS the agent` という設計思想は一貫している
- `yfinance` ベースで低コスト運用できている
- `research / decision / monitor / review` の流れは理解しやすい
- `research_history.json`、`portfolio.csv`、`monitor_history.json` など、意思決定ログを残す基盤がある
- 実際の売買履歴とレポートが蓄積されており、改善余地を検証できる状態に入っている

### まだ弱い点

- 運用に必須な通知経路と、読み取り専用経路の責務分離が甘い
- CSV / JSON のデータ契約が揃っていない
- 提案の妥当性を検証するコードと、実約定を集計するコードが混ざっている
- 週次レビューや校正レポートが「参考情報」で止まっていて、次の判断に十分返っていない
- B枠（paper portfolio）があるのに、まだ仮説検証ループとして立ち上がっていない

## 重要な指摘

### 1. Slack が送信時以外でも必須になっている

`investor/config.py` で `slack_webhook_url` を必須設定として即時評価しているため、`decision_agent.py` を import しただけで `.env` 前提になる。

この影響で、次のような経路まで Slack 設定がないと壊れる。

- `/decision` の読み取り専用経路
- `--paper` の paper portfolio 記録経路
- 別ディレクトリからの import / 実行

これは「通知機能の依存」が「全体の依存」になっている状態。

### 2. 1銘柄 25% 上限が実コードでは守られていない

README では「1銘柄最大配分 25%」とあるが、`validate_proposals()` は今回の新規注文サイズしか見ていない。  
既存保有の同一 ticker がある状態で追加 BUY してもブロックされない。

そのため、ルール上は分散投資でも、実際には同一銘柄へ積み増しできてしまう。

### 3. データ契約のズレが静かに残っている

今回見つかったズレの例:

- `entry_price_range` の en dash と hyphen の扱いが処理系で違う
- `portfolio.csv` は `mfe_pct` なのに、統計側では `mfe_capture_pct` を読んでいる箇所がある
- paper portfolio と real portfolio でスキーマが揃っていない
- closed row に必要な項目が欠けても、一部の集計が無言でスキップする

この種のズレはすぐクラッシュしないが、**レビュー指標を少しずつ信用できなくする**。

### 4. weekly review の指標名と実態がずれている

`decision_history.json` に残しているのは「提案した BUY / PASS」であり、実際に人間が執行した約定ではない。  
それなのに `weekly_review.py` では `BUY実行` と表示している。

これはヒューマン・イン・ザ・ループ設計と噛み合っていない。

正しくは次のどちらかに分ける必要がある。

- 提案数を出すレビュー
- 実約定数を出すレビュー

### 5. プロダクトの本当の価値がまだ明文化されていない

戦略レビューの観点では、`investor/` の価値は「AIが銘柄を勧めること」ではなく、  
**自分の投資判断を継続的に校正できること**にある。

いまの構造だと debate、watchlist、paper、review が全部あるのに、  
それらが「どの判断を次回どう変えるか」に十分つながっていない。

## 戦略レビューで整理できたこと

### いま再定義すべき目的

現時点の `investor/` は、次のどちらを主目的にするかをはっきり決めたほうがいい。

1. 毎日売買提案を出す運用ツール
2. 判断精度を高める校正エンジン

レビュー結果から見ると、長期的に価値が出るのは 2。

理由:

- すでに判断ログが蓄積されている
- 実運用履歴があり、仮説検証に入れる
- 同じ市場環境が続かない以上、単発の勝率より校正能力のほうが価値が高い

### 推奨する戦略の順番

#### 戦略A: 校正エンジン化

主役にするもの:

- `research_history.json`
- `portfolio.csv`
- `decision_history.json`
- `weekly_review.py`
- `show_calibration_stats.py`

狙い:

- 確信度が本当に当たっているかを見る
- signal ごとの alpha を出す
- 保有日数、利確タイミング、PASS 判断の質を見直す
- 次回の sizing や PASS 条件に返す

#### 戦略B: 実弾運用ツールとしての信頼性強化

必須の守り:

- Slack は送信時のみ必須にする
- 同一 ticker の追加 BUY を制御する
- atomic write を入れる
- CSV スキーマをそろえる
- silent skip を減らす

これは差別化ではなく土台。

#### 戦略C: 仮説検証マシン化

次段階として有効:

- A枠 real と B枠 paper の並走
- watchlist からの昇格ルールの比較
- 強気局面 / 弱気局面ごとの戦略差分
- HOLD_CASH / NO_TRADE の判断品質の評価

面白い方向だが、A と B を固める前に広げると運用が散る。

## いま優先すべきアクション

### 最優先

1. Slack 依存を通知経路に限定する
2. `validate_proposals()` に既存保有込みの 25% 制約を入れる
3. `weekly_review.py` の `BUY実行` 表記を見直す
4. `portfolio.csv` / `paper_portfolio.csv` / 統計処理の列名を統一する

### 次にやるべきこと

1. `show_calibration_stats.py` を「見るだけ」から「次の判断に使う」位置へ上げる
2. `decision_history.json` を提案ログと実約定ログで分ける
3. B枠を動かして real vs paper の差分を週次で出す
4. closed / open row のバリデーションを追加して silent skip を減らす

### いまは後回しでよいこと

1. ダッシュボードや見た目の強化
2. 新しい debate ペルソナの追加
3. watchlist UI の作り込み
4. 高度な最適化やリアルタイム化

## このプロジェクトをどう見るべきか

いまの `investor/` は「完成度の高い投資エージェント」ではなく、  
**実運用ログを伴った、かなり良い投資判断実験基盤** と見るのが正確。

その意味では、今回のレビューで見つかった違和感は悲観材料ではない。  
むしろ「どこを直せばこの基盤が強くなるか」がかなり明確になった。

ポイントは次の通り。

- debate を増やすより、記録の精度を上げる
- 新機能を足すより、提案と実約定を分ける
- 勝率を見るより、SPY alpha と再現性を見る
- AI の賢さより、データ契約と運用の信頼性を優先する

## まとめ

今回のレビューでわかったことはシンプル。

`investor/` は方向性自体は良い。  
ただし今の課題は「もっと賢い分析をすること」ではなく、  
**判断の記録、検証、反映のループをきれいに閉じること** にある。

したがって、次のテーマは機能拡張ではなく次の3つ。

1. 信頼境界を直す
2. データ契約をそろえる
3. 校正ループをプロダクトの中心に据える

これが整理できると、`investor/` は単なる提案ツールから、  
自分専用の投資判断OSに近づく。
